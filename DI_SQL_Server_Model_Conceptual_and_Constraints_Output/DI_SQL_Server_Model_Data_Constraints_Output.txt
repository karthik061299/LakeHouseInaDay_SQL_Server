1. DATA EXPECTATIONS
========================================================================

Data Expectations define the quality standards, completeness requirements, accuracy levels, format specifications, and consistency rules that data must meet to ensure reliable reporting and analytics.


1.1 DATA COMPLETENESS EXPECTATIONS
------------------------------------------------------------------------

1. Employee Resource Data Completeness:
   • GCI Identifier (gci id, GCI_ID) must be populated for all employee records
   • First Name and Last Name are mandatory for all active employees
   • Social Security Number (SSN) must be present for all W2 employees
   • Job Title must be assigned to all active consultants
   • Employee Status must be defined for all records
   • Start Date is required for all employee assignments
   • Employee Type classification is mandatory
   • Email Address must be provided for active employees

2. Client Organization Data Completeness:
   • Client Code must be unique and populated for all clients
   • Client Name is mandatory for all client records
   • Client Type classification is required
   • Client Sector must be defined for reporting purposes
   • At least one contact method (phone or email) must be available

3. Project Engagement Data Completeness:
   • Project Name must be unique and descriptive
   • Client Code association is mandatory
   • Project Start Date is required
   • Project City and Project State must be populated for location-based reporting
   • Project Type classification is mandatory
   • Billing Type must be defined (Hourly, Fixed Price, etc.)

4. Time Entry Data Completeness:
   • GCI Identifier (GCI_ID) is mandatory for all time entries
   • Period End Date (PE_DATE) must be populated
   • Task Identifier (task_id, Task_Id) linking to project assignment is required
   • At least one time category (ST, OT, DT, TIME_OFF, HO, Sick_Time) must have non-zero hours
   • Billable flag must be set for all time entries

5. Project Assignment Data Completeness:
   • GCI Identifier linking to employee is mandatory
   • Assignment Start Date is required
   • Bill Rate Standard Time (bill st, HR_Project_ST) must be defined for billable assignments
   • Pay Rate Standard Time (pay st) is required for cost calculations
   • Assignment Status must be maintained
   • Client Code and Project Name associations are mandatory

6. Hiring Workflow Data Completeness:
   • Track Identifier (TrackID) must be unique for each workflow instance
   • Process Identifier (Process_ID) is mandatory
   • Level Identifier (Level_ID) must track current workflow stage
   • Initiator Name and Initiator Email are required
   • Workflow Status must be maintained throughout the process
   • Date Created timestamp is mandatory

7. Financial Rate Structure Data Completeness:
   • Bill Rate and Pay Rate must be defined with corresponding units
   • Markup Percentage must be calculated and validated
   • Gross Profit (GP) and Gross Profit Margin (GPM) must be computed
   • Rate effective dates must be maintained


1.2 DATA ACCURACY EXPECTATIONS
------------------------------------------------------------------------

1. Financial Calculations Accuracy:
   • Gross Profit must equal (Bill Rate × Hours) - (Pay Rate × Hours)
   • Gross Profit Margin must equal (Gross Profit / Revenue) × 100
   • Markup Percentage must equal ((Bill Rate - Pay Rate) / Pay Rate) × 100
   • Net Bill Rate (NBR) calculations must be accurate after adjustments
   • Derived Revenue must match (Bill Rate × Billable Hours)

2. Time Calculations Accuracy:
   • Total hours per day should not exceed 24 hours
   • Weekly hours should align with business day calculations
   • Approved hours must match or be less than submitted hours
   • Billable hours must be validated against project billing rules
   • Expected Hours calculations must consider Working Days from calendar

3. Date Calculations Accuracy:
   • Termination Date (termdate, term_date) must be >= Start Date
   • Final End Date (Final_End_date) must be >= Assignment End Date
   • Period End Date (PE_DATE) must align with Week Date for timesheet reporting
   • Assignment duration calculations must be accurate
   • Consultant Ageing (Cons_Ageing) must reflect actual days on assignment

4. Headcount Metrics Accuracy:
   • Begin HC + Starts - Terms - Other Project Ends - OffBoard = End HC
   • Net Addition = End HC - Begin HC
   • Headcount movements must reconcile across reporting periods
   • Voluntary Terminations (Vol_term) must be subset of total Terms

5. Rate Validation Accuracy:
   • Bill Rate must be >= Pay Rate for positive margin
   • Actual Markup must be <= Maximum Allowed Markup
   • Submitted Bill Rate must be validated against client agreements
   • Rate changes must be tracked with effective dates
   • Months in Same Rate (Mons_in_Same_Rate) must be calculated correctly


1.3 DATA FORMAT EXPECTATIONS
------------------------------------------------------------------------

1. Identifier Formats:
   • GCI Identifier: varchar(50) - alphanumeric unique identifier
   • Client Code: varchar(50) - standardized client identification code
   • SSN: varchar(50) - formatted as XXX-XX-XXXX or encrypted
   • Task Identifier: numeric(18,9) - numeric project task reference
   • Track Identifier: varchar(50) - unique workflow tracking code

2. Date and Time Formats:
   • All date fields: datetime format (YYYY-MM-DD HH:MM:SS)
   • Period End Date: datetime - typically Saturday end of week
   • Week Date: datetime - week ending date for aggregation
   • Year-Month Code (YYMM, st_yymm): int format YYYYMM for period identification

3. Monetary Formats:
   • Bill Rate, Pay Rate: money or float with 2 decimal precision
   • Gross Profit (GP): money format with currency precision
   • Salary: money format with appropriate currency
   • All financial amounts must be non-negative unless representing adjustments

4. Percentage Formats:
   • Markup, GPM: numeric with 2 decimal places (e.g., 25.50 for 25.5%)
   • Region percentages: varchar(10) or numeric format
   • All percentages should be stored as actual percentage values (not decimals)

5. Text Field Formats:
   • Names (First Name, Last Name): varchar(50) - proper case formatting
   • Email addresses: varchar(50-100) - valid email format with @ symbol
   • Phone numbers: varchar(50) - standardized format with area code
   • Status fields: varchar - predefined enumeration values

6. Hours Formats:
   • Time hours (ST, OT, DT): float with 2 decimal precision
   • Expected Hours: real or float format
   • All hour values must be non-negative


1.4 DATA CONSISTENCY EXPECTATIONS
------------------------------------------------------------------------

1. Cross-Table Consistency:
   • GCI Identifier must be consistent across New_Monthly_HC_Report, SchTask, Timesheet_New, report_392_all
   • Client Code must match between New_Monthly_HC_Report and report_392_all
   • Task Identifier must be consistent across Timesheet_New and report_392_all
   • Period End Date must align across all timesheet-related tables

2. Status Consistency:
   • Employee Status (Emp_Status, employee_status, Status) must be synchronized
   • Assignment Status (asstatus, status) must reflect current state
   • Workflow Status must align with Level Identifier progression
   • Terminated employees must have Termination Date populated

3. Rate Consistency:
   • Bill Rate units (bill st units) must match across all references
   • Pay Rate units (pay st units) must be consistent
   • Rate changes must be reflected with proper effective dates
   • Offshore rates must be distinguished from onshore rates

4. Organizational Consistency:
   • Business Unit (BU), Sub Business Unit (SBU), Department (Dept) hierarchy must be valid
   • Reporting relationships (Manager, Recruiting Manager, Resource Manager) must be active employees
   • Location assignments must be consistent across employee and project records

5. Temporal Consistency:
   • Start Date must be <= End Date for all assignments
   • Time entries must fall within assignment date ranges
   • Workflow dates must progress logically through stages
   • Historical data must be preserved with proper effective dating


1.5 DATA TIMELINESS EXPECTATIONS
------------------------------------------------------------------------

1. Time Entry Timeliness:
   • Timesheets must be submitted within 3 business days of period end
   • Approved hours must be finalized within 5 business days of submission
   • Consultant hours and Approved hours must be reconciled weekly

2. Hiring Workflow Timeliness:
   • Response Time must be tracked from requisition received to submission
   • Interview dates must be scheduled within defined SLA timeframes
   • Workflow completion must be tracked with DateCompleted timestamp

3. Financial Data Timeliness:
   • Bill rates must be updated before assignment start date
   • Gross Profit calculations must be current as of reporting period
   • Backlog Revenue and Backlog GP must be updated monthly

4. Headcount Reporting Timeliness:
   • Headcount movements must be recorded in the period they occur
   • Begin HC and End HC must be calculated for each reporting month
   • Terminations and offboarding must be recorded on actual dates

2. CONSTRAINTS
========================================================================

Constraints define the technical and business rules that enforce data integrity, ensure valid relationships, and maintain data quality standards.


2.1 PRIMARY KEY CONSTRAINTS
------------------------------------------------------------------------

1. Employee Resource Entity:
   • Primary Key: GCI Identifier (gci id, GCI_ID)
   • Constraint: Must be unique and not null
   • Table Reference: New_Monthly_HC_Report.id (numeric identity), report_392_all.id

2. Client Organization Entity:
   • Primary Key: Client Code (client code)
   • Constraint: Must be unique and not null
   • Table Reference: Derived from New_Monthly_HC_Report, report_392_all

3. Project Engagement Entity:
   • Primary Key: Composite of Project Name + Client Code
   • Constraint: Combination must be unique
   • Table Reference: Derived from Hiring_Initiator_Project_Info.Project_Name

4. Time Entry Entity:
   • Primary Key: Composite of GCI_ID + PE_DATE + task_id
   • Constraint: Combination must be unique to prevent duplicate time entries
   • Table Reference: Timesheet_New (gci_id, pe_date, task_id)

5. Hiring Workflow Entity:
   • Primary Key: Track Identifier (TrackID)
   • Constraint: Must be unique and not null
   • Table Reference: SchTask.TrackID

6. Project Assignment Entity:
   • Primary Key: Composite of GCI Identifier + Task Identifier + Start Date
   • Constraint: Combination must be unique
   • Table Reference: report_392_all.Task_Id, Hiring_Initiator_Project_Info

7. Working Calendar Entity:
   • Primary Key: Composite of Year + Month
   • Constraint: Must be unique for each calendar period
   • Table Reference: Working_Days_2025 ([Year], [Month])


2.2 FOREIGN KEY CONSTRAINTS
------------------------------------------------------------------------

1. Time Entry to Employee Resource:
   • Foreign Key: Timesheet_New.gci_id
   • References: Employee Resource.GCI Identifier
   • Constraint: Must reference valid active employee
   • Action: Restrict deletion of employee with time entries

2. Time Entry to Project Assignment:
   • Foreign Key: Timesheet_New.task_id
   • References: Project Assignment.Task Identifier
   • Constraint: Must reference valid project assignment
   • Action: Restrict deletion of assignment with time entries

3. Project Assignment to Employee Resource:
   • Foreign Key: report_392_all.[gci id]
   • References: Employee Resource.GCI Identifier
   • Constraint: Must reference valid employee
   • Action: Cascade updates to GCI Identifier

4. Project Assignment to Project Engagement:
   • Foreign Key: report_392_all.proj_ID or Hiring_Initiator_Project_Info.Project_Name
   • References: Project Engagement.Project Name
   • Constraint: Must reference valid project
   • Action: Restrict deletion of project with active assignments

5. Project Engagement to Client Organization:
   • Foreign Key: New_Monthly_HC_Report.[client code]
   • References: Client Organization.Client Code
   • Constraint: Must reference valid client
   • Action: Cascade updates to client code

6. Hiring Workflow to Employee Resource:
   • Foreign Key: SchTask.GCI_ID
   • References: Employee Resource.GCI Identifier
   • Constraint: Must reference valid employee or candidate
   • Action: Restrict deletion during active workflow

7. Time Entry to Working Calendar:
   • Foreign Key: Timesheet_New.pe_date
   • References: Working Calendar.Period End Date
   • Constraint: Must reference valid business period
   • Action: Restrict deletion of calendar periods with time entries


2.3 UNIQUE CONSTRAINTS
------------------------------------------------------------------------

1. Employee Resource:
   • Social Security Number (SSN) must be unique across all employees
   • Collabera Email (Collabera_Email_ID) must be unique
   • Oracle Person Identifier (OraclePersonID, ORC_ID) must be unique

2. Client Organization:
   • Client Code must be unique
   • DNB Number (client DNB) must be unique if populated

3. Project Engagement:
   • Netsuite Project Identifier (NetsuiteProjectId) must be unique
   • MS Project Identifier (Ms_ProjectId, MS_ProjectId) must be unique
   • FP Project Identifier (FP_Proj_ID) must be unique for fixed price projects

4. Hiring Workflow:
   • Track Identifier (TrackID) must be unique
   • Requisition Number (HR_Recruiting_ReqNo) must be unique per requisition

5. Project Assignment:
   • Combination of GCI Identifier + Task Identifier + Start Date must be unique
   • No overlapping assignments for same employee on same project


2.4 NOT NULL CONSTRAINTS
------------------------------------------------------------------------

1. Employee Resource Mandatory Fields:
   • GCI Identifier (gci id, GCI_ID) - NOT NULL
   • First Name (first name, FName) - NOT NULL
   • Last Name (last name, LName) - NOT NULL
   • Employee Status (Emp_Status, employee_status) - NOT NULL
   • Employee Type (employee type) - NOT NULL
   • Start Date (start date) - NOT NULL

2. Client Organization Mandatory Fields:
   • Client Code (client code) - NOT NULL
   • Client Name (client name) - NOT NULL
   • Client Type (client_type) - NOT NULL

3. Project Engagement Mandatory Fields:
   • Project Name (Project_Name) - NOT NULL
   • Client Code (client code) - NOT NULL
   • Project Start Date (HR_Project_StartDate, start date) - NOT NULL
   • Project City (project city) - NOT NULL
   • Project State (project state) - NOT NULL
   • Project Type (project_type, Project_Type) - NOT NULL

4. Time Entry Mandatory Fields:
   • GCI Identifier (gci_id, GCI_ID) - NOT NULL
   • Period End Date (pe_date, PE_DATE) - NOT NULL
   • Task Identifier (task_id, Task_Id) - NOT NULL
   • Billable Flag (BILLABLE) - NOT NULL

5. Hiring Workflow Mandatory Fields:
   • Track Identifier (TrackID) - NOT NULL
   • Process Identifier (Process_ID) - NOT NULL
   • Level Identifier (Level_ID) - NOT NULL with DEFAULT 0
   • Last Level (Last_Level) - NOT NULL with DEFAULT 0
   • Date Created (DateCreated) - NOT NULL

6. Project Assignment Mandatory Fields:
   • GCI Identifier - NOT NULL
   • Task Identifier - NOT NULL
   • Assignment Start Date - NOT NULL
   • Bill Rate Standard Time - NOT NULL for billable assignments
   • Assignment Status - NOT NULL

7. Working Calendar Mandatory Fields:
   • Year - NOT NULL
   • Month - NOT NULL
   • Working Days (Working_Days) - NOT NULL
   • Holiday counts (Holiday_Baroda, Holiday_US, Holiday_Chennai) - NOT NULL


2.5 CHECK CONSTRAINTS
------------------------------------------------------------------------

1. Date Validation Constraints:
   • Start Date <= End Date for all assignments
   • Termination Date (termdate, term_date) >= Start Date
   • Final End Date (Final_End_date) >= Assignment End Date
   • PO Start Date (po start date) <= PO End Date (po end date)
   • Period End Date must be a Saturday (week ending date)
   • Interview dates must be >= Requisition Received Date

2. Financial Validation Constraints:
   • Bill Rate (bill st, Bill_ST, BillRate) > 0 for billable assignments
   • Pay Rate (pay st) > 0 for all assignments
   • Bill Rate >= Pay Rate (to ensure positive margin)
   • Gross Profit (GP) = (Bill Rate × Hours) - (Pay Rate × Hours)
   • Gross Profit Margin (GPM) >= 0 and <= 100
   • Markup Percentage >= 0
   • Actual Markup <= Maximum Allowed Markup
   • Net Bill Rate (NBR, Net_Bill_Rate) > 0

3. Time Entry Validation Constraints:
   • Standard Time Hours (ST) >= 0
   • Overtime Hours (OT) >= 0
   • Double Time Hours (DT) >= 0
   • Total daily hours <= 24
   • Approved hours <= Submitted hours
   • Sick Time Hours (Sick_Time) >= 0
   • Time Off Hours (TIME_OFF) >= 0
   • Holiday Hours (HO) >= 0

4. Headcount Validation Constraints:
   • Begin HC (Begin HC) >= 0
   • End HC (End HC) >= 0
   • Starts - New Project >= 0
   • Starts - Internal movements >= 0
   • Terms >= 0
   • Other project Ends >= 0
   • OffBoard >= 0
   • Voluntary Terminations (Vol_term) <= Total Terms
   • Net Addition = End HC - Begin HC

5. Status Validation Constraints:
   • Employee Status must be in ('Active', 'Terminated', 'On Leave', 'Suspended')
   • Assignment Status must be in ('Active', 'Completed', 'On Hold', 'Cancelled')
   • Workflow Status must be in ('In Progress', 'Completed', 'Pending', 'Rejected')
   • Billable Flag must be in ('Yes', 'No') or bit (0, 1)

6. Percentage Validation Constraints:
   • Region1 Percentage + Region2 Percentage = 100 (if both populated)
   • Markup Percentage >= 0 and <= 200
   • Gross Profit Margin >= -100 and <= 100

7. Identifier Validation Constraints:
   • GCI Identifier must be alphanumeric
   • SSN must follow format XXX-XX-XXXX (if not encrypted)
   • Email addresses must contain @ symbol
   • Phone numbers must be numeric with optional formatting


2.6 DATA TYPE CONSTRAINTS
------------------------------------------------------------------------

1. Numeric Data Types:
   • ID fields: numeric(18,0) or numeric(18,9) for precision
   • GCI Identifier: varchar(50) or int depending on table
   • Money fields: money or decimal(18,2) for financial data
   • Hours: float or real for decimal precision
   • Percentages: numeric or float for calculations

2. String Data Types:
   • Names: varchar(50) for first name, last name
   • Descriptions: varchar(100-8000) or nvarchar(max) for long text
   • Codes: varchar(50) for identifiers and codes
   • Email: varchar(50-100) for email addresses
   • Status: varchar(25-50) for status fields

3. Date/Time Data Types:
   • All date fields: datetime for date and time storage
   • Timestamp fields: timestamp or datetime for audit trails

4. Boolean Data Types:
   • Flag fields: bit (0/1) or varchar(3) ('Yes'/'No')
   • Examples: IS_SOW, IS_Offshore, Billable, IsBillRateSkip

5. Large Text Data Types:
   • Job descriptions: nvarchar(max) or text
   • Comments: varchar(8000) or text
   • Notes: text or nvarchar(max)


2.7 REFERENTIAL INTEGRITY CONSTRAINTS
------------------------------------------------------------------------

1. Employee-Assignment Integrity:
   • All project assignments must reference valid active employees
   • Terminated employees cannot have new assignments created
   • Employee termination date must be >= last assignment end date

2. Client-Project Integrity:
   • All projects must be associated with valid clients
   • End Client Code must reference valid client if populated
   • Client relationships must be maintained throughout project lifecycle

3. Assignment-Time Entry Integrity:
   • Time entries must fall within assignment date ranges
   • Time entries cannot exist for terminated assignments
   • Task Identifier in time entries must reference valid assignments

4. Workflow-Assignment Integrity:
   • Hiring workflows must result in valid project assignments
   • Track Identifier must link workflow to assignment
   • Workflow completion must precede assignment start date

5. Calendar-Time Entry Integrity:
   • Period End Date must exist in Working Calendar
   • Time entries must align with valid business periods
   • Holiday hours must correspond to defined holidays in calendar

6. Rate-Assignment Integrity:
   • Bill rates must be effective for assignment period
   • Rate changes must be tracked with effective dates
   • Historical rates must be preserved for audit purposes

7. Organizational Hierarchy Integrity:
   • Manager references must point to valid active employees
   • Reporting relationships must not create circular references
   • Business unit hierarchy must be valid and consistent


2.8 BUSINESS LOGIC CONSTRAINTS
------------------------------------------------------------------------

1. Assignment Overlap Constraints:
   • An employee cannot have overlapping assignments on different projects without approval
   • Full-time employees cannot exceed 40 hours per week across all assignments
   • Part-time employees must have hours limited per assignment terms

2. Rate Change Constraints:
   • Bill rate increases require client approval
   • Bill rate decreases must be documented with reason
   • Rate changes must maintain minimum margin requirements
   • Months in Same Rate must be recalculated on rate changes

3. Termination Constraints:
   • Termination Date must be populated when Employee Status = 'Terminated'
   • Termination Reason must be provided for all terminations
   • Offboarding Date must be >= Termination Date
   • Final End Date must be calculated considering all extensions

4. Billing Constraints:
   • Billable hours cannot exceed approved hours
   • Non-billable hours must have valid reason codes
   • Overtime must be pre-approved for billable projects
   • Double time requires special authorization

5. Hiring Workflow Constraints:
   • Workflow must progress through defined levels sequentially
   • Level Identifier cannot skip levels
   • Workflow cannot be completed without reaching Last Level
   • Client Interview Required flag must be honored

6. Financial Calculation Constraints:
   • Gross Profit must be recalculated on rate or hour changes
   • Derived Revenue must match billed amounts
   • Backlog calculations must exclude completed assignments
   • ACA costs must be applied based on eligibility rules

3. BUSINESS RULES
========================================================================

Business Rules define the operational policies, processing logic, transformation guidelines, and decision criteria that govern how data is created, updated, and used within the system.


3.1 EMPLOYEE MANAGEMENT BUSINESS RULES
------------------------------------------------------------------------

1. Employee Onboarding Rules:
   • New employees must complete hiring workflow before assignment creation
   • GCI Identifier must be assigned during onboarding process
   • Employee Type must be determined before start date (W2, C2C, 1099, FTE)
   • Visa Type must be verified for non-citizens before assignment
   • Background checks must be completed before client assignment
   • I9 documentation must be approved before start date

2. Employee Status Rules:
   • Employee Status transitions: Active → On Leave → Active or Active → Terminated
   • Terminated employees cannot be reactivated without new hiring workflow
   • On Leave status requires return date specification
   • Suspended status requires HR approval and documented reason

3. Employee Classification Rules:
   • Employee Category must align with Employee Type
   • Standard Job Title must be assigned based on skills and experience
   • Experience Level Title must be determined by years of experience and role
   • HR Band Level and HR Band Title must be consistent with grade
   • Role Family and Sub Role Family must follow organizational taxonomy

4. Employee Termination Rules:
   • Termination Date must be set when employment ends
   • Termination Reason must be documented (Voluntary, Involuntary, Project End, etc.)
   • Offboarding workflow must be initiated within 24 hours of termination
   • Final End Date must account for notice period and extensions
   • Access revocation must occur on or before Offboarding Date
   • Exit interviews required for voluntary terminations

5. Employee Replacement Rules:
   • Replacement hires must reference original employee GCI Identifier
   • Replacement Reason must be documented
   • Knowledge transfer period should be planned if possible
   • Replacement must have comparable or better skill match

6. Employee Redeployment Rules:
   • Internal movements must be tracked separately from new hires
   • Redeployment Led By must be documented
   • Existing Resource flag must be set to 'Yes'
   • Previous assignment must be closed before new assignment starts


3.2 CLIENT AND PROJECT MANAGEMENT BUSINESS RULES
------------------------------------------------------------------------

1. Client Onboarding Rules:
   • Client Code must be assigned before project creation
   • Client Type must be classified (Direct, MSP, VMS, Subtier)
   • Client Sector must be defined for industry reporting
   • Master Service Agreement must be in place before assignments
   • Client Manager contact information must be maintained

2. Project Creation Rules:
   • Project Name must be unique within client
   • Project Type must be defined (Time & Material, Fixed Price, SOW)
   • Project Start Date must be >= current date for new projects
   • Project Location (City, State, Country) must be specified
   • Billing Type must be determined (Hourly, Daily, Weekly, Monthly, Fixed)
   • Purchase Order must be received before assignment start

3. Project Classification Rules:
   • IS_SOW flag must be set for Statement of Work projects
   • IS_Offshore flag must be set if project involves offshore resources
   • ITSS classification must be applied for IT services projects
   • FP_TM indicator must distinguish Fixed Price from Time & Material
   • Practice Type must be assigned (Application Development, Infrastructure, etc.)
   • Vertical and Tower must be defined for service line reporting

4. Project Billing Rules:
   • Billable projects must have Bill Rate defined before time entry
   • Non-billable projects must have Non Billing Type specified
   • Overtime Allowed flag must be set based on client agreement
   • Double Time Allowed flag requires special client authorization
   • 9 Hours Allowed flag and effective date must be documented
   • Overtime Billable flag determines if OT is charged to client

5. Project Location Rules:
   • Project City and Project State must be valid geographic locations
   • Project County must be specified for tax calculation purposes
   • Project Zone must be assigned for regional reporting
   • Region 1 and Region 2 percentages must sum to 100% if split billing
   • Remote work arrangements must be documented in project details

6. End Client Rules:
   • End Client must be identified if different from direct client
   • End Client Sector must be captured for industry analysis
   • MSP Name must be recorded for managed service provider arrangements
   • Subtier information must be complete for multi-tier engagements


3.3 TIME ENTRY AND TIMESHEET BUSINESS RULES
------------------------------------------------------------------------

1. Time Entry Submission Rules:
   • Timesheets must be submitted weekly by Period End Date (Saturday)
   • Time entries must be submitted within 3 business days of period end
   • Consultant must submit hours before manager approval
   • Time entries cannot be submitted for future dates
   • Retroactive time entries require manager approval

2. Time Category Rules:
   • Standard Time (ST): Regular working hours up to 40 per week
   • Overtime (OT): Hours beyond 40 per week (if allowed)
   • Double Time (DT): Hours on holidays or special circumstances (if allowed)
   • Time Off: Paid time off, vacation, personal days
   • Holiday (HO): Company-recognized holidays
   • Sick Time: Illness-related absences
   • Non-Billable categories: Time not charged to client

3. Time Approval Rules:
   • Timesheet Manager must approve hours within 5 business days
   • Approved hours cannot exceed submitted hours without explanation
   • Rejected timesheets must be resubmitted with corrections
   • Approved Standard Time, Approved Overtime, Approved Double Time must be tracked separately
   • Client approval may be required for overtime hours

4. Billable vs Non-Billable Rules:
   • Billable flag must be set based on project billing arrangement
   • Training time is typically non-billable
   • Internal meetings are non-billable unless client-facing
   • Bench time (unassigned) is non-billable
   • Administrative tasks are non-billable

5. Time Validation Rules:
   • Total daily hours cannot exceed 24 hours
   • Weekly hours should not exceed 60 without special approval
   • Time entries must fall within assignment date range
   • Time entries for terminated employees require special handling
   • Expected Hours vs Actual Hours variance must be monitored

6. Working Days Calculation Rules:
   • Business Days must be calculated from Working Calendar
   • Holidays must be excluded from billable day calculations
   • Location-specific holidays must be applied (US, Baroda, Chennai)
   • Week Cycle must be defined for timesheet period determination


3.4 FINANCIAL AND BILLING BUSINESS RULES
------------------------------------------------------------------------

1. Bill Rate Determination Rules:
   • Bill Rate Standard Time must be defined in client agreement
   • Bill Rate Overtime typically = Bill Rate ST × 1.5
   • Bill Rate Double Time typically = Bill Rate ST × 2.0
   • Offshore rates (Bill Rate Standard Time Offshore) may differ from onshore
   • Bill Rate Units must be specified (Hourly, Daily, Weekly, Monthly, Annual)
   • FP Bill Rate applies to fixed price projects
   • SCA Hourly Bill Rate applies to Service Contract Act covered positions

2. Pay Rate Determination Rules:
   • Pay Rate Standard Time must be defined before assignment
   • Pay Rate Units must match billing frequency where applicable
   • Loaded Pay Rate includes base pay plus benefits and overhead
   • Salary Amount and Salary Units must be defined for salaried employees
   • Candidate Salary represents offer amount
   • Pay Rate to Consultant is actual compensation paid

3. Markup and Margin Calculation Rules:
   • Markup Percentage = ((Bill Rate - Pay Rate) / Pay Rate) × 100
   • Actual Markup must be calculated from actual rates
   • Maximum Allowed Markup must not be exceeded without approval
   • Submitted Markup is proposed margin in sales process
   • Client Markup Percentage may be pre-negotiated in MSA
   • Markup validation must occur before assignment approval

4. Gross Profit Calculation Rules:
   • Gross Profit (GP) = Revenue - Direct Costs
   • Gross Profit = (Bill Rate × Billable Hours) - (Pay Rate × Total Hours)
   • Gross Profit Margin (GPM) = (Gross Profit / Revenue) × 100
   • GP2020 and GPM2020 represent calculations using 2020 methodology
   • Submitted Gross Profit Margin is proposed margin in sales
   • Negative GP must be flagged for review

5. Revenue Recognition Rules:
   • Derived Revenue = Bill Rate × Approved Billable Hours
   • Revenue must be recognized in the period time is worked
   • Fixed Price revenue recognized based on milestone completion
   • Backlog Revenue represents future committed revenue
   • Backlog Gross Profit calculated from backlog revenue

6. Cost Allocation Rules:
   • ACA Cost (Affordable Care Act) must be allocated based on eligibility
   • ACA Classification determines benefit cost allocation
   • Benefits Plan costs must be included in loaded rates
